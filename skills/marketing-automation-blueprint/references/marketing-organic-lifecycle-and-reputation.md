# Marketing Organic, Lifecycle, And Reputation

## 1. Organic content, SEO, and AEO

Build a topic and intent graph from real user jobs:

```text
intent/problem -> useful canonical asset -> proof/source -> product action
-> related questions -> locale/channel transformations -> freshness/retirement
```

Requirements:

- canonical URLs, metadata, structured data, internal links, sitemaps and accessibility;
- original utility, evidence and product fit; no low-value scaled-content spam;
- query/answer coverage by language/market, not keyword substitution;
- exact claims, authorship/provenance, update/expiry and correction path;
- landing/deep-link availability, startup and product expectation parity;
- search/social/video variants derive from canonical content manifests.

`organic-1` — Agent volume is not user value. Automatically stop or consolidate content that is duplicative, unsupported, stale, harmful, or has no distinct job.

## 2. Social, community, creators, and partners

For every publishing surface define:

```text
audience and product truth:
content/creative manifest and rights:
platform adapter/current policy:
cadence/frequency and conversation purpose:
reply/moderation/support escalation:
sponsorship/affiliate/AI disclosure:
recipient action and deep-link fallback:
measurement, holdout and fatigue:
```

Creator/affiliate contracts need audience fit, deliverables, claims, disclosure, usage rights, brand safety, payment/commission source, fraud/quality controls, content retention, correction/termination, and incremental measurement.

Community automation may classify, answer verified FAQs, route support, moderate and summarize. It must not invent users, astroturf, manufacture consensus, argue with criticism, or impersonate organic advocacy.

## 3. Lifecycle orchestration

```text
product event
-> current user/account state
-> consent, age, territory, entitlement and suppression
-> next-best valuable action
-> channel, quiet-hour and frequency policy
-> holdout
-> delivery and deep-link recovery
-> product outcome and countermetrics
-> model/policy update
```

Separate transactional, security, support, social, educational, lifecycle and promotional messages. An opt-out or channel denial changes routing; it does not justify bypassing consent through another channel.

Return messages should expose a real unfinished job, changed state, collaborator action, new relevant capability, saved continuity, or requested reminder. Do not fabricate urgency, friend activity, expiring value, or guilt.

## 4. Cross-promotion

Build a versioned portfolio-fit graph:

```text
source product/surface -> audience/job edge -> target product/value
-> claim IDs -> eligibility/consent/age/territory
-> disclosed placement/frequency/dismissal
-> universal/app/web link and store fallback
-> activation/retained-value attribution -> holdout -> cannibalization
```

Product design owns placement semantics; marketing owns target selection, message, frequency and measurement; campaign/referral specialists own any offer or reward.

Never force another-product installation, block core use, disguise a house ad as system UI, collect contacts without need/consent, or apply one billing/link path to regions with different rules.

## 5. Update adoption and benefits

Update communication distinguishes:

- security/compatibility minimum version;
- flexible feature release and education;
- migration or service change;
- incident compensation;
- optional celebration/reward after verified valuable adoption.

Reward contract:

```text
signed source/target version:
eligible account and value event:
authoritative idempotent grant, cap, retry and reversal:
economy/entitlement effect:
support/fraud/rollback:
```

Never reward a rating/review, permission, ad click, spend, forced referral/contact upload, or another-app install. Do not use reward pressure to conceal a broken compatibility plan.

## 6. Ratings, reviews, feedback, and reputation

Consume `review-solicitation-policy` for public request eligibility and
`product-feedback-learning-loop` for private intake, authorized review
ingestion, response evidence and product learning; never invent one universal
cross-platform prompt.

- Apple/Google may provide standardized native review requests with platform-controlled presentation/limits.
- Steam states developers should not request reviews inside the application.
- Platform rules and APIs change; every adapter has a current authority record.
- Eligibility is a neutral meaningful-value event where permitted, never predicted positivity, spend or vulnerability.
- Private feedback and support remain independently available to everyone.
- No review/rating rewards, five-star imagery/steering, sentiment gates, suppression, fake reviews or indiscriminate auto-replies.

Reputation operations ingest authorized review data, classify verified issues, link them to support/product candidates, and respond only when a factual clarification, recovery path or acknowledgement is useful. Do not argue, reveal personal information, ask for a higher rating, or amplify every review mechanically.
